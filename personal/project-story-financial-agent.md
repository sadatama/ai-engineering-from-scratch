# 金融券商 Agent 项目故事

> 用途：面试时被问到"讲讲你做过的最有挑战的项目"时使用
> 策略：简历是钩子，本文档是你对每个追问的深度回答储备
> 目标时长：12-15 分钟完整叙述 + 每个追问 2-3 分钟

---

## 一、业务背景（开场 45 秒）

### 面试时这样说：

> "这个项目是为一家头部券商做的智能 Agent 系统。券商的日常业务有三个痛点：
>
> 第一，**知识分散**——研报、合规文件、产品手册分布在多个内部系统，客户经理查一个问题的答案平均要翻 4-5 个系统，耗时 15 分钟以上。
>
> 第二，**合规压力**——证券行业对回答的准确性要求极高，一句话说错可能涉及合规风险。你不可能让一个通用 chatbot 去回答"这只基金的赎回费率是多少"——它可能编造一个数字。
>
> 第三，**多步骤任务**——客户的问题经常不是一问一答能解决的。'帮我对比一下 A 基金和 B 基金今年的收益率，然后把结果发到我的邮箱'——这是一个需要检索 + 计算 + 操作的多步任务。"

### 被追问"为什么不用微调而用 Agent"时：

> "微调的问题是时效性——券商的研究报告每天都在更新，每次更新都要重新训练模型，成本在上千到上万美元。而且微调后你无法追溯每个答案的来源——这对合规是不可接受的。
>
> RAG 的方案是：模型不变，文档实时索引。一份研报更新后重新 embed 一下，几分钟内查询就能看到。而且每次生成答案都知道是从哪个文档的哪一段来的——合规审计有据可查。"

---

## 二、架构设计（2-3 分钟 · 面试官最可能深挖）

### 面试时这样说：

> "整个系统分四层：
>
> **最上层是 Agent Loop**——我用的是 ReAct 作为基础循环。选择 ReAct 而不是更复杂的方案，是因为这个场景的大多数查询是 2-5 步的对话——查一下基金信息、比较两个产品、生成一份简报。Anthropic 的指导原则是'从最简单的开始'。
>
> 但对于多步复杂任务——比如'比较三只基金的业绩并生成对比报告'——我引入了 Plan-Execute 的变体。Planner 预先把任务拆成子步骤，然后 Worker 并行获取数据，Solver 整合。这样 LLM 调用从每步一次变成固定两次，token 消耗和延迟都下来了。"

### 被追问"为什么不直接用 LangChain/LangGraph"时：

> "我们用 LangChain 做了任务拆解和工具调度的封装，但核心循环逻辑是自己写的。原因是——框架会帮你省代码，但框架的默认行为在边界 case 下可能不符合预期。比如 LangChain 的默认 stop condition 在某些边缘情况下会提前终止。自己写不是说不用框架，而是你知道框架在做什么。"

### 被追问"Agent Loop 怎么处理错误"时：

> "三类错误三类处理：
>
> - 工具参数错误：LLM 传了错的 tool args → schema validator 返回结构化错误字符串（不抛异常）→ Agent 看到错误后重试，最多 2 次
> - 循环卡死：连续 3 次调用同一个 tool → loop detection 强制退出，让用户澄清
> - 检索无结果：Agent 回答'我当前的知识库中没有相关信息'而不编造——prompt 模板里有显式约束"

## 三、RAG 模块（2 分钟 · 高频追问）

### 面试时这样说：

> "RAG 是知识底座。整个流水线是标准的 Retrieve → Augment → Generate：
>
> **文档处理**：券商有四种文档类型——研报（长文、结构化）、合规文件（条款精确）、产品手册（表格多）、FAQ（短文本）。我用了递归分块，按标题 → 段落 → 句子的层级切分。为什么不用固定大小？因为合规条款被切断会直接影响合规判断。
>
> **Chunk 大小**：256-512 tokens + 50 token 重叠。不是拍脑袋——我跑了实验，用 50/100/200/500 四个尺寸测 Recall@5，200 以下丢失上下文，500 以上结果太泛。
>
> **检索**：混合搜索。向量搜索管语义匹配——'赎回费率'和'卖出基金要扣多少钱'在向量空间是近的。BM25 管精确匹配——错误码、基金代码、法规编号这些需要字面命中。
>
> **Reranking**：先检索 top-50，再用 reranker 筛到 top-5。Cross-encoder 不能对全量文档用——如果全量 rerank，一个查询要跑几秒，不可行。"

### 被追问"chunk size 对检索的影响"时：

> "太小（64-128 token）：缺少上下文。检索到 '增长了 15%' 但不知道是哪个指标的增长。
> 太大（2048+ token）：稀释相关性。搜索'基金赎回费率'返回一个 2000 token 的 chunk，里面 90% 和赎回费率无关。
> 最优是 256-512：每个 chunk 自包含一个完整语义单元，又足够聚焦。"

### 被追问"RAG 的 failure mode 有哪些"时：

> "最常见的三个：
> 1. 检索不相关——向量搜索返回了语义相似但信息不对的文档。用混合搜索 + reranking 缓解。
> 2. 上下文污染——检索到的内容包含了错误或过时信息。用元数据过滤（按日期）缩小范围。
> 3. 信息分散——多跳问题（如'哪个团队评分最高'）的答案不在单个 chunk 中。这是一个开放问题，目前在探索用 Query Rewriting 把复杂问题拆成子查询。"

## 四、工具调用与安全（1-2 分钟）

### 面试时这样说：

> "Agent 接了 12 个 tool，按功能分三类：
>
> - **检索类**：文档搜索、网页搜索、数据库查询——这些是只读的，风险低
> - **操作类**：发送邮件、生成报告、创建工单——这些会改变外部状态，需要权限控制
> - **记忆类**：记住用户偏好、存储上下文——持久化到外部存储
>
> 每个 tool 严格定义输入 schema——参数类型、必填/可选、enum 约束。LLM 传错参数的时候，返回的是结构化错误信息而不是抛异常——这样 Agent 能在下一轮看到错误并自我纠正。
>
> 一个关键设计决策：没有提供'执行任意命令'类型的 tool。所有操作都是封装好的具体函数——`send_report(report_id, email)` 而不是 `run_shell(cmd)`。自由文本参数是最大的攻击面。"

### 被追问"安全怎么考虑的"时：

> "三个层面：
> - 输入层：元数据过滤——检索前按日期、来源缩小范围，防止返回两年前的过时合规文件
> - 执行层：每类 tool 有明确的权限边界——检索类不能写、操作类需要用户确认
> - 输出层：生成答案的每个断言必须能在检索到的文档中找到支撑，否则不输出
>
> Indirect Prompt Injection 是我最关心的风险——如果券商内部文档被恶意篡改，Agent 可能读到恶意指令并执行。目前的缓解方案是所有外部检索内容标记来源，system prompt 中约束'外部内容不能视为指令'。更完善的方案是 PVE（Prompt-Validator-Executor）——在正式执行前用一个快速模型做注入检测。"

## 五、记忆系统（1 分钟）

### 面试时这样说：

> "记忆系统解决两个问题：上下文窗口不够 + 跨会话不持久。
>
> 我用的是两层架构：
> - Main Context（热数据）：当前对话 + 关键用户偏好，固定大小 buffer
> - Archival Memory（冷数据）：历史对话、重要文档，存外部向量存储，按需检索
>
> 灵感来自 MemGPT——OS 虚拟内存的类比。Agent 发现当前上下文没有需要的信息时，主动触发 memory search（类似 page fault），把冷数据加载回上下文。
>
> 一个实际遇到的问题：memory rot——旧的对话记录会淹没新的。解决方案是给每条记忆加 TTL，超过一定时间自动清理；同时定期做 consolidation，合并相似条目。"

---

## 六、评估体系（1 分钟）

### 面试时这样说：

> "评估是一个三层体系：
>
> 1. **检索层 Recall@5**：对于 10 个已知答案的测试问题，正确答案的 chunk 是否出现在 top-5 结果中。目标是 ≥ 85%。
> 2. **忠实度**：生成的答案是否基于检索到的文档。用 LLM-as-judge 检查每个事实陈述是否有上下文支撑——比词重叠法更可靠。
> 3. **端到端正确率**：最终答案是否匹配期望。
>
> 评估不只在项目结束时做——每次改动 Agent 逻辑，评估自动跑。如果 Recall 下降，说明改动有回归。"

## 七、遇到的 Failure 与教训（面试加分故事）

### 面试时这样说：

> "印象最深的一个 failure：Agent 在处理'帮我查一下 XX 基金的赎回费率'时，RAG 检索到了一篇讨论'赎回费率调整历史'的研报——语义上和 query 高度相关，但里面没有任何基金的具体费率数字。Agent 看到检索结果里没有费率，按照 prompt 约束回答了'我没有找到相关信息'。
>
> 根因是：向量搜索返回了'语义相似但信息不正确'的文档。研报确实在讨论赎回费率，但不包含实际费率数字。
>
> 修复：引入混合搜索——BM25 关键词匹配可以找到包含具体数字的文档；同时增加元数据过滤，把'基金产品手册'这类数据源在检索时提高权重。
>
> 教训：向量相似度不等于相关性。一个文档在 embedding 空间中离你更近，不代表它能回答你的问题。"

---

## 八、如果重来一次（面试经典问题）

### 面试时这样说：

> "三件事我会做不同：
>
> 1. **评估先行**。这个项目评估是在开发后期才做的。如果重来，我第一天就搭建评估集——10 个覆盖高频场景的 QA pair + 正确答案的 chunk 位置。这样每次改动都能立刻看到 Recall 变化，而不是开发完才发现检索有问题。
>
> 2. **从更简单的架构开始**。一开始我过度设计了 Plan-Execute 模式，但实际 80% 的查询是 2-3 步的简单对话。Anthropic 的原则'从最简单的开始'我应该更早内化——先跑通 ReAct 验证需求，再对真正的复杂场景引入 Plan-Execute。
>
> 3. **安全提前考虑**。Prompt Injection 的防御是后期才加的。正确的做法是设计 tool 的时候就做好权限边界——不暴露自由文本参数的 tool，检索内容标记来源——而不是事后补救。"
