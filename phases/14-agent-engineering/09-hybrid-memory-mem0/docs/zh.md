# 混合记忆：向量 + 图 + KV（Mem0）

> Mem0 (Chhikara et al., 2025) 将记忆视为三个并行存储——向量用于语义相似度，KV 用于快速事实查找，图用于实体关系推理。一个评分层在检索时融合三者的结果。这是 2026 年外部记忆的生产级标准。

**类型：** 构建
**语言：** Python (stdlib)
**前置条件：** Phase 14 · 07 (MemGPT)、Phase 14 · 08 (Letta Blocks)
**时间：** 约 75 分钟

## 学习目标

- 解释为什么单一存储（纯向量、纯图、纯 KV）对 agent 记忆来说是不够的。
- 列举 Mem0 的三个并行存储，以及它们各自优化的目标。
- 描述 Mem0 的融合评分——相关性、重要性、新鲜度——以及为什么它是加权求和而非分层结构。
- 使用标准库实现一个玩具级三存储记忆系统，包含向所有三个存储写入的 `add()` 和融合结果的 `search()`。

## 问题

单一存储对以下三类查询中总有一类是不适用的：

- **语义相似度** —— "我们上周讨论了关于 agent 漂移的什么内容？" 向量存储胜出；KV 和图会遗漏。
- **事实查找** —— "用户的电话号码是多少？" KV 存储胜出；向量存储浪费，图存储过度设计。
- **关系推理** —— "哪些客户共享同一个账单实体？" 图存储胜出；向量和 KV 无法回答。

生产级 agent 在一个会话中会发出所有三种类型的查询。单一存储的记忆总是在其中两类上是错误的。Mem0 的贡献在于将三者连接在单一的 `add`/`search` 接口之后，并通过评分函数融合它们。

## 核心概念

### 三个并行存储

Mem0 (arXiv:2504.19413, April 2025) 在 `add(text, user_id, metadata)` 时：

1. 从文本中提取候选事实（LLM 驱动的步骤）。
2. 将每个事实写入向量存储（embedding）以支持语义搜索。
3. 将每个事实按 (user_id, fact_type, entity) 键写入 KV 存储以支持 O(1) 查找。
4. 将每个事实以类型边写入图存储 (Mem0g) 以支持关系查询。

在 `search(query, user_id)` 时：

1. 向量存储通过 embedding 余弦相似度返回 top-k。
2. KV 存储返回基于 (user_id, type, entity) 查询派生的精确命中。
3. 图存储返回从查询实体可达的子图。
4. 评分层融合三者的结果。

### 融合评分

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **相关性** —— 向量余弦相似度、KV 精确匹配、图路径权重。
- **重要性** —— 在写入时标注或学习获得（某些事实更重要：姓名、ID、策略）。
- **新鲜度** —— 自上次写入或读取以来的指数衰减。

权重根据产品进行调整。聊天 agent 调高 `w_recency`；合规 agent 调高 `w_importance`；检索 agent 调高 `w_relevance`。

### Mem0g 与时间推理

Mem0g 添加了冲突检测器。当新事实与现有边存在矛盾时，现有边被标记为无效但不删除。时间查询（"用户三月份住哪个城市？"）遍历特定时间点的有效子图。

这是 Letta 的失效模式泛化的合规级行为。

### 基准数字

Mem0 论文报告（2025 年）：

- **LoCoMo**（长对话记忆）：91.6
- **LongMemEval**（长程情景记忆）：93.4
- **BEAM 1M**（1M token 记忆基准）：64.1

比较基线（全上下文 128k LLM、纯向量存储、纯 KV 存储）全部落后 10+ 分。仅凭基准数据不能证明选择合理——运维形态才是关键——但这些数字表明融合设计并非偶然的误差范围。

### 作用域分类

Mem0 按作用域划分记忆：

- **用户记忆** —— 跨会话持久化，按 `user_id` 键控。
- **会话记忆** —— 在一个线程内持久化。
- **Agent 记忆** —— 每个 agent 实例的状态。

每次写入选择一个作用域。检索可以跨作用域查询，每个作用域有不同的权重。不经过深思熟虑就混合作用域，会导致"助手告诉 Alice 关于 Bob 项目的事"这类事故。

### 这种模式可能出错的地方

- **Embedding 漂移。** 在前一百个查询上看起来正确的向量结果，随着语料库增长而退化。对最常用的 N 条记录进行定期重新 Embedding。
- **KV schema 膨胀。** `(user_id, type, entity)` 看起来简单，直到每个团队都添加自己的 `type`。每季度审计 type 集合。
- **图爆炸。** 一个嘈杂的提取器每条消息添加 50 条边。限制每次 `add` 调用的图写入量；丢弃低置信度的边。

## 动手实现

`code/main.py` 使用标准库实现了三存储模式：

- `VectorStore` —— 简单的 token 重叠相似度作为 embedding 的替代。
- `KVStore` —— 基于 `(user_id, fact_type, entity)` 键控的字典。
- `GraphStore` —— 类型边 (subject, relation, object, valid)。
- `Mem0` —— 顶层外观，包含 `add()`、`search()`、融合评分和作用域感知检索。
- 一个在多用户、多会话对话上的完整追踪演示。

运行：

```
python3 code/main.py
```

输出展示三个独立的召回路径以及融合后的 top-k。在 `main()` 顶部调整评分权重，观察排名的变化。

## 实际应用

- **Mem0 (Apache 2.0)** —— 生产就绪。使用 Postgres + Qdrant + Neo4j 自托管，或使用托管云服务。
- **Letta** —— 三层 core/recall/archival；自行选择向量和图后端。
- **Zep** —— 商业替代方案，具有时序知识图谱和事实提取能力。
- **自定义构建** —— 当你需要对提取器（合规场景）或融合权重（新鲜度占主导的语音 agent）进行精确控制时。

## 练习

1. 用真实的 embedding 模型（sentence-transformers、Ollama、OpenAI embeddings）替换玩具向量相似度。在合成长对话上测量 recall@10。在 1000 次写入后排名是否会出现漂移？
2. 添加时间查询：`search(query, as_of=timestamp)`。只返回在该时间点或之前有效的记录。哪个存储需要最多的改动？
3. 实现冲突检测器：如果传入事实与图边冲突，使旧边失效并记录两者。测试 "用户住在北京" -> "用户住在上海"。
4. 将融合评分器扩展为包含 `user_feedback` 维度（对检索记录的点赞）。如何防止作弊（agent 只返回它已经点赞过的记录）？
5. 阅读 Mem0 文档 (`docs.mem0.ai`)。将玩具实现移植为 `mem0` 客户端调用。在相同的 20 个测试查询上比较检索质量。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|---------|
| 混合记忆 | "向量加图加 KV" | 三个存储并行写入，检索时融合 |
| 事实提取 | "记忆摄取" | LLM 步骤，将文本分解为 (entity, relation, fact) 元组 |
| 融合评分 | "相关性排序" | 相关性、重要性、新鲜度的加权求和 |
| 作用域 | "记忆命名空间" | user / session / agent —— 决定谁能看到什么 |
| Mem0g | "记忆图" | 具有时间有效性的类型边，用于关系查询 |
| 时间失效 | "软删除" | 标记矛盾的边为无效；从不删除 |
| Embedding 漂移 | "检索腐化" | 随着语料库增长，向量质量退化；定期重新 Embedding |
