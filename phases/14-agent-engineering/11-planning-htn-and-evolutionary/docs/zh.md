# 使用 HTN 与进化搜索进行规划

> 符号化规划处理需要可证明正确性的场景。进化代码搜索处理适应度函数可机器验证的场景。ChatHTN (2025) 和 AlphaEvolve (2025) 展示了当与 LLM 配对时，每种方法能解锁什么能力。

**类型：** 构建
**语言：** Python (stdlib)
**前置条件：** Phase 14 · 02 (ReWOO 和 Plan-and-Execute)
**时间：** 约 75 分钟

## 学习目标

- 解释层级任务网络：任务、方法、操作符、前置条件、效果。
- 描述 ChatHTN 的混合循环——符号化搜索配合 LLM 回退分解。
- 解释 AlphaEvolve 的进化循环，以及为什么它只在具有程序化评估器时才有效。
- 使用标准库实现一个玩具级 HTN 规划器加一个玩具级进化搜索。

## 问题

ReWOO（Lesson 02）、Plan-and-Execute 和 ReAct 覆盖了大多数 agent 规划场景。有两类场景它们处理得不好：

1. **需要可证明正确性的规划。** 调度、飞行路径、合规工作流——规划必须在构造上就正确无误。一个流畅的 LLM 规划如果偶尔虚构步骤，是不可接受的。
2. **具有可机器验证适应度函数的优化。** 矩阵乘法、调度启发式、编译器 pass——目标不是"一个正确的规划"，而是"最佳的规划"。

HTN 规划和 AlphaEvolve 分别解决这两个不同的问题。两者都将 LLM 作为放大器而非替代品。

## 核心概念

### 层级任务网络（HTN）

一个 HTN 包含：

- **任务** —— 复合任务（待分解）和基本任务（可直接执行）。
- **方法** —— 将复合任务分解为子任务的方式，并带有前置条件。
- **操作符** —— 具有前置条件和效果的基本动作。
- **状态** —— 一组事实。

规划：给定一个目标任务和一个初始状态，找到一系列基本操作符的分解，使其每一步的前置条件在序列中都能被满足。

HTN 比 LLM 更古老，至今仍是可证明正确规划的参考标准。

### ChatHTN (Gopalakrishnan et al., 2025)

ChatHTN (arXiv:2505.11814) 将符号化 HTN 与 LLM 查询交织：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有方法适用，询问 LLM："在状态 `s` 下，你会如何分解 `task`？"
3. 将 LLM 回复转化为候选子任务。
4. 根据操作符 schema 验证；拒绝无效的分解。
5. 递归。

论文的核心主张：每一个生成的计划都是可证明正确的，因为 LLM 建议仅作为候选分解进入系统，从不作为直接的规划编辑。符号化层拥有正确性的控制权；LLM 扩展了方法库。

在线方法学习 (OpenReview `gwYEDY9j2x`, 2025 后续) 添加了一个学习器，通过回归泛化 LLM 生成的分解——将 LLM 查询频率降低高达 75%。

### AlphaEvolve (Novikov et al., 2025)

AlphaEvolve (arXiv:2506.13131, DeepMind, June 2025) 是一种不同的事物：由 Gemini 2.0 Flash/Pro 组合编排的进化代码搜索。

循环：

1. 从一个种子程序 + 一个程序化评估器（返回适应度分数）开始。
2. LLM 组合提议变更（变异）。
3. 通过评估器运行变异。
4. 保留最佳结果；再次变异。

已发表的成果：

- 56 年来首次改进 Strassen 4x4 复数矩阵乘法（48 次标量乘法）。
- 通过 Borg 调度启发式恢复 Google 0.7% 的计算资源。
- 在前沿工作负载上 FlashAttention 加速 32%。

硬性约束：适应度函数必须是可机器验证的。对文本答案进行进化搜索不会收敛。

### 何时使用哪种方法

| 问题类别 | 使用 | 原因 |
|---------|-----|------|
| 具有硬约束的调度 | HTN + ChatHTN | 可证明的正确性 |
| 编译器优化 | AlphaEvolve | 可机器验证的适应度 |
| 多步任务执行 | ReAct / ReWOO | LLM 在循环中，无需形式保证 |
| 有测试的代码改进 | AlphaEvolve | 测试即是评估器 |
| 策略限定的自动化 | HTN | 前置条件编码策略 |

### 这种模式可能出错的地方

- **没有操作符的 HTN。** 没有前置条件/效果 schema，正确性主张就会崩溃。ChatHTN 的 "LLM 建议分解" 需要 schema 来拒绝无效方案。
- **没有真实评估器的 AlphaEvolve。** "问 LLM 代码是否更好" 不是一个适应度函数。评估器必须是确定性的且速度很快。
- **过度工程化。** 大多数 agent 任务不需要这两种方法。首先考虑 ReAct 或 ReWOO。

## 动手实现

`code/main.py` 实现两个玩具：

- 一个标准库 HTN 规划器，包含操作符、方法、前置条件、效果，以及当没有方法匹配复合任务时启动的 `LLMFallback`。这里的 "LLM" 是一个脚本化的分解器，因此规划器可以离线运行。
- 一个标准库进化搜索，搜索算术程序：生长表达式，使其输出在测试集上最小化 `|f(x) - target|`。评估器是确定性的。

运行：

```
python3 code/main.py
```

追踪输出展示 HTN 规划器分解一个复合任务（中间有一次 LLM 回退）以及进化循环收敛到一个目标表达式。

## 实际应用

- **HTN 规划器** —— `pyhop`、`SHOP3`，或为特定领域的策略强制构建自己的规划器。
- **ChatHTN** —— 研究代码；该模式（符号化 + LLM 回退）可以清晰地移植到任何 HTN 规划器。
- **AlphaEvolve** —— DeepMind 论文；该模式（组合 + 评估器）是可复现的。OpenEvolve 和类似的开源分支正在涌现。
- **Agent 框架** —— 目前还没有内置一流 HTN 或 AlphaEvolve 支持的框架。将其构建为 subagent 或后台 worker。

## 交付物

`outputs/skill-hybrid-planner.md` 生成一个混合规划器框架（HTN 或进化），LLM 的角色被明确界定。

## 练习

1. 为 HTN 规划器扩展回溯功能：当某个操作符的结果条件在运行时失败时，回滚并尝试下一个方法。
2. 为 ChatHTN 添加 LLM 方法缓存：当 LLM 在状态模式 `P` 中分解任务 `T` 时，存储结果。下次调用时首先重新检查方法库。
3. 将进化搜索的评估器替换为真实的测试套件。进化一个能通过 20 个测试用例的排序函数；报告收敛所需的代数。
4. 阅读 AlphaEvolve 的评估器设计说明。为你关心的领域设计一个评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5. 组合使用：用 HTN 将复合任务分解为子任务，然后对每个子任务的基本操作符使用进化搜索。它在什么地方表现出色，在什么地方过度工程化？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|---------|
| HTN | "层级规划器" | 具有操作符、前置条件、效果的任务分解 |
| 方法 | "分解规则" | 将复合任务分解为子任务的方式 |
| 操作符 | "基本动作" | 具有前置条件和效果的具体步骤 |
| ChatHTN | "LLM + HTN" | 符号化规划器在没有方法匹配时询问 LLM |
| AlphaEvolve | "进化代码搜索" | 组合 LLM 变异代码；确定性评估器选择最佳结果 |
| 适应度函数 | "评估器" | 对输出进行确定性、可机器验证的评分 |
| 在线方法学习 | "缓存 LLM 分解" | 存储并泛化 LLM 规划以降低查询成本 |
