# 故障模式：Agent 为什么会出问题

> MASFT（Berkeley，2025）将 14 种多 Agent 故障模式分为 3 个类别。Microsoft 的分类法记录了现有的 AI 故障如何在 Agent 场景中被放大。业界现场数据归纳出五种反复出现的模式：幻觉操作、范围蔓延、级联错误、上下文丢失、工具误用。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 第十四阶段 · 05（Self-Refine 与 CRITIC），第十四阶段 · 24（可观测性）
**时间：** 约 60 分钟

## 学习目标

- 说出 MASFT 的三个故障类别以及每个类别中至少四种具体模式。
- 解释为什么 Agent 故障会放大现有的 AI 故障模式（偏见、幻觉）。
- 描述五种业界反复出现的模式及其缓解措施。
- 使用标准库实现一个检测器，为 Agent 执行轨迹打上故障模式标签。

## 问题

团队交付的 Agent 在 90% 的执行轨迹上运行正常。那 10% 的故障并非随机噪声——它们属于少数几个反复出现的类别。一旦你能识别它们，就能监控并修复它们。

## 概念

### MASFT（Berkeley，arXiv:2503.13657）

多 Agent 系统故障分类法（Multi-Agent System Failure Taxonomy）。14 种故障模式分为 3 个类别。标注者间 Cohen's Kappa 系数为 0.88——这些类别是可靠可区分的。

核心主张：故障是多 Agent 系统中的根本设计缺陷，而非可以通过更好的基础模型修复的 LLM 局限性。

### Microsoft Agent AI 系统故障模式分类法

- 现有的 AI 故障（偏见、幻觉、数据泄露）在 Agent 场景中被放大。
- 自主性带来了新的故障：规模化下的非预期操作、工具误用、任务漂移。
- 该白皮书是 Agent 产品的风险登记册。

### 表征 Agent AI 中的故障（arXiv:2603.06847）

- 故障源于编排、内部状态演化和环境交互。
- 不仅仅是"糟糕的代码"或"糟糕的模型输出"。

### LLM Agent 幻觉综述（arXiv:2509.18970）

两种主要表现形式：

1. **指令遵循偏离**——Agent 未遵循系统提示词。
2. **长程上下文误用**——Agent 遗忘或错误应用来自早期轮次的上下文。

子意图错误：遗漏（跳过步骤）、冗余（重复步骤）、乱序（步骤顺序错误）。

### 五种业界反复出现的模式

Arize、Galileo、NimbleBrain 在 2024-2026 年的现场分析归纳出：

1. **幻觉操作。** Agent 调用不存在的工具或编造参数。
2. **范围蔓延。** Agent 将任务扩展到超出用户要求（创建额外的 PR、发送额外的邮件）。
3. **级联错误。** 一个错误的调用触发下游连锁反应。一个幻影 SKU 幻觉触发四个 API 调用——一次跨系统事故。
4. **上下文丢失。** 长周期任务遗忘早期轮次的约束条件。
5. **工具误用。** 用错误的参数调用正确的工具，或完全调用错误的工具。

级联错误是最致命的。Agent 无法区分"我失败了"和"任务不可能完成"，常常在 400 错误上幻觉出一个成功消息来关闭循环。

### 缓解：每一步都设门控

在推理链的每一步设置自动化验证门控，检查事实是否基于环境状态。具体而言：

- 每步安全分类器（第 21 课）。
- 工具调用参数验证（第 06 课）。
- 将检索到的内容与已知事实交叉检查（第 05 课，CRITIC）。
- 通过重新探测状态来检测成功幻觉（文件是否确实被创建了？）。

### 故障监控可能出错的地方

- **只标记崩溃。** 大多数 Agent 故障产生的输出看起来是有效的。需要内容级别的检查。
- **没有基线。** 漂移检测需要最后一个已知良好状态；没有它，你就无法判断"情况正在恶化"。
- **过度告警。** 每个故障都产生一个告警。需要聚类和速率限制。

## 构建

`code/main.py` 使用标准库实现故障模式标注器：

- 一个覆盖五种模式的合成执行轨迹数据集。
- 每种模式的检测函数（工具调用、输出、重复操作的特征模式）。
- 一个标注器，为每条轨迹打标签并报告模式分布。

运行：

```
python3 code/main.py
```

输出：每条轨迹的标签 + 聚合分布，这是 Phoenix 执行轨迹聚类功能的一个低成本复现。

- **Langfuse** 用于会话回放 + 标注。

## 练习

1. 添加一个"成功幻觉"检测器：Agent 返回成功但目标状态未改变。
2. 为你构建的产品标注 100 条真实执行轨迹。哪种模式占主导？修复它的成本是多少？
3. 实现"级联半径"指标：给定第 N 步的故障，它影响了多少下游步骤？
4. 阅读 MASFT 的 14 种故障模式。选出三种适用于你产品的模式。编写检测器。
5. 将一个检测器接入 CI 任务：如果 >=5% 的执行轨迹被标记为某种模式，则构建失败。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|----------------|------------------------|
| MASFT | "多 Agent 故障分类法" | Berkeley 的 14 种模式分类 |
| Cascading error（级联错误） | "连锁故障" | 一个早期错误通过 N 步传播 |
| Context loss（上下文丢失） | "忘记约束条件" | 长周期轮次丢失早期轮次的事实 |
| Tool misuse（工具误用） | "错误的工具/错误的参数" | 有效调用，但调用了错误的工具或参数 |
| Success hallucination（成功幻觉） | "伪造完成" | Agent 在 400 错误上声称成功；状态未改变 |
| Scope creep（范围蔓延） | "越权操作" | Agent 做了超出要求的事 |
| Instruction-following deviation（指令遵循偏离） | "不服从" | 忽略系统提示词或用户约束 |
| Sub-intention errors（子意图错误） | "计划缺陷" | 计划执行中的遗漏、冗余、乱序 |
