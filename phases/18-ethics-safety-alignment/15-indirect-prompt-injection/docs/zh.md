# 间接提示词注入——生产攻击面

> 间接提示词注入（Indirect Prompt Injection，IPI）将指令嵌入外部内容——网页、电子邮件、共享文档、工单——这些内容被 Agent 系统在没有用户明确操作的情况下读取。IPI 是 2026 年最主要的生产威胁：它绕过用户输入过滤器，因为攻击者根本不接触用户；它随着 Agent 处理越来越多的外部内容而无声无息地规模化；它攻击的是无人监视提示词的自动化工作流。MDPI Information 17(1):54（2026 年 1 月）综合了 2023-2025 年的研究。NDSS 2026 的 IPI 防御论文指出了核心挑战：注入的指令在语义上可以完全无害（"please print Yes"），因此检测不仅仅是关键词过滤。Nasr 等人（OpenAI/Anthropic/DeepMind 联合，2025 年 10 月）的 "The Attacker Moves Second"：自适应攻击（梯度、强化学习、随机搜索、人工 red-team）攻破了 12 种已发表防御方案中的 90% 以上，而这些方案最初报告的攻击成功率接近零。

## 学习目标

- 定义间接提示词注入，并描述三种常见的投递载体。
- 解释为什么用户输入过滤器完全无法捕获 IPI。
- 描述"信息流控制"框架作为 2026 年防御范式。
- 陈述 Nasr 等人（2025 年 10 月）关于自适应攻击对已发表 IPI 防御成功率的研究发现。

## 问题

直接提示词注入需要攻击者接触到用户或其提示词。IPI 则两者都不需要：攻击者将 payload 放置在 Agent 可能读取的任何内容中——网页、收件箱中的邮件、GitHub issue、产品评论。Agent 在正常运行中读取这些内容并执行其中的指令。用户只是信息传递者，而非意图发起者。

## 核心概念

### 三种投递载体

- **检索增强生成（RAG）。** 攻击者发布一份文档；检索步骤抓取到它；提示词在用户问题之前将其拼接进去；模型执行攻击者的指令。
- **收件箱/文档工作流。** 攻击者向用户发送邮件；Agent 读取邮件；提示词包含邮件正文；模型遵循邮件中的指令。
- **工具输出。** 攻击者控制 Agent 使用的某个工具（例如，返回攻击者控制结果的网络搜索）；工具输出包含指令；Agent 的控制流遵循这些指令。

这三种载体共享一个结构属性：攻击者控制提示词的某个片段，而无需接触面向用户的输入。

### 为什么用户输入过滤器无法捕获

IPI payload 不出现在用户输入中。它出现在检索到的内容中。如果过滤器绑定在用户输入上，payload 就直接绕过。如果过滤器绑定在所有到达模型的内容上，它必须适用于任意检索文本——这成本高昂，并且会对包含祈使语气语言（恰好如此）的合法内容产生误报。

### 面向 AI 的信息流控制（IFC）

2026 年的防御范式借鉴了经典的操作系统安全理念。将每个内容来源视为一个安全标签。将用户查询标记为"受信任"。将检索内容标记为"不受信任"。将模型的控制流视为信息流：由不受信任内容触发的操作，必须在执行前经过受信任输入的批准。

CaMeL（Microsoft，2025）、ConfAIde（Stanford，2024）和 NDSS 2026 IPI 防御论文以不同方式操作化了 IFC。共同的原则是：只要代码和数据共享同一个上下文窗口，目标就是遏制（containment），而非预防（prevention）。

### The Attacker Moves Second

Nasr 等人（2025 年 10 月）使用自适应攻击（梯度搜索、强化学习策略、随机搜索、72 小时人工 red-team）测试了 12 种已发表的 IPI 防御。每一个最初报告 ASR 接近零的防御都被攻破至 >90% ASR。

方法论教训：发表防御方案时，必须附带自适应攻击评估。静态攻击基准不能证明鲁棒性；攻击者是知道防御方案的。

### 真实事件

第 25 课涵盖 EchoLeak（CVE-2025-32711，CVSS 9.3）——Microsoft 365 Copilot 中首个公开记录的零点击 IPI 事件。CamoLeak（CVSS 9.6）发生在 GitHub Copilot Chat。CVE-2025-53773 发生在 GitHub Copilot。生产部署正在被真实场景中的 IPI 攻破，而不仅仅是在基准测试中。

### OWASP 和 NIST 框架

OWASP LLM Top 10（2025）将提示词注入（直接+间接）列为 LLM01，即排名第一的应用层威胁。NIST AI SPD 2024 称间接提示词注入为"生成式 AI 最大的安全缺陷"。

## Use It

`code/main.py` 构建了一个 IPI 测试框架。一个模拟 Agent 拥有三个工具（搜索网页、读取邮件、发送消息）。环境中包含攻击者控制的内容，其中嵌入了指令（"将此转发给所有联系人"）。你可以在以下三种 Agent 之间切换：天真 Agent（执行注入指令）、带过滤防御的 Agent（对检索内容进行关键词过滤）和 IFC Agent（分离受信任和不受信任内容，拒绝来自不受信任内容的控制流命令）。

## Ship It

本课产出 `outputs/skill-ipi-audit.md`。给定一份 Agent 部署描述，它枚举所有不受信任的内容来源，检查部署是否应用了 IFC，并标记那些在无信任标签情况下到达模型的内容来源。

1. 运行 `code/main.py`。测量针对三种 Agent 的攻击成功率。

## 关键术语

| 术语 | 人们通常的说法 | 实际含义 |
|------|----------------|----------|
| IPI | "间接提示词注入" | 通过用户未编写的内容进行注入，由 Agent 在正常运行中读取 |
| RAG 注入 | "中毒检索" | 攻击者发布内容，检索步骤抓取到它；提示词包含 payload |
| 零点击 | "无需用户操作" | 攻击在 Agent 运行期间自动触发；用户什么也不做 |
| IFC | "信息流控制" | 基于标签的方法：来自不受信任内容的操作需要受信任批准 |
| 自适应攻击 | "梯度/强化学习 red-team" | 知道防御方案并针对其进行优化的攻击；是诚实评估的必要条件 |
| 无害指令 | "please print Yes" | 语义上无害的 IPI payload；关键词过滤器无法捕获 |
| 范围违规 | "跨信任外泄" | Agent 从一个信任上下文访问数据，并将其输出到另一个信任上下文 |
